DQL简单查询

查询一个字段

select 字段名 from 表名;

查询多个字段使用 “,” 隔开

select 字段名1,字段名2 from 表名;

查询所有字段

select * from 表名;(效率低，可读性差，实际开发不建议)
select 字段名1,字段名2,字段名3... from 表名;

给查询的列起别名(使查询结果显示的字段名为别名)

select 字段名 as 别名 from 表名;
select 字段名 别名 from 表名;
select 字段名 '别名' from 表名; (别名是中文或有空格，用单引号)
select 字段名 "别名" from 表名; (不推荐，单引号是标准)

查询时可以对字段使用数学表达式

select sal*12 as yearsal from 表名;

DQL 条件查询

语法

条件查询有哪些条件

  1. = 等于
    查询薪资等于 800 的员工姓名和编号
    select empno,ename from emp where sal = 800;
    查询SIMTH的编号和薪资
    select empno,sal from emp where ename = 'smith';
  2. <> 或 != 不等于
  3. <,>,<=,>=
  4. between … and … 等价于 >= and <=
    查询薪资在2450和3000之间的员工信息（包含2450和3000） 注意：使用 between and 需要遵循左小右大
  5. is/is not
    查询哪些员工的津贴/补助(不)为 null
    select empno,ename,comm from emp where comm is null;
    select empno,ename,comm from emp where comm is not null;
    注意：在数据库中需要使用 is 在衡量 null
  6. and 并且
    查询工作岗位是 MANAGER 且工资大于 2500 的员工信息
    select empno,ename,job,sal from emp where job = 'manager' and sal > 2500;
  7. or 或者
    查找工作岗位是 MANAGER 和 SALESMAN 的员工
    select empno,ename,job,sal from emp where job = 'MANAGER' or job = 'SALESMAN';
    注意：and 的优先级比 or 高，查找 工资大于2500 且 部门编号为10或20 的所有员工
    select * from emp where sal > 2500 and (depto = 10 or depto = 20);
  8. in 包含(相当于多个 or)
    查找工作岗位是 MANAGER 和 SALESMAN 的员工
    select empno,ename,job,sal from emp where job in ('MANAGER', 'SALESMAN');
  9. not in 不包含

模糊查询

  1. % 表示任意多个字符
  2. _ 表示任意一个字符

例:
查询名字含有 ‘o’ 的员工
select ename from emp where ename like '%o%';

查询名字第二个字母是 ‘a’ 的员工
select ename from emp where ename like '_a%';

查询名字有 ‘_’ 的员工
select ename from emp where ename like '%\_%';

排序

查询所有员工的薪资，排序(默认升序)
select ename,sal from emp order by sal;

查询所有员工的薪资，降序
select ename,sal from emp order by sal desc;

查询所有员工的薪资，排序(指定升序)
select ename,sal from emp order by sal asc;

查询员工名字和薪资，薪资升序，如果薪资一样按名字升序
select ename,sal from emp order by sal asc, ename asc;

查询员工薪资，排序(根据字段位置，即第二列)(不推荐)
select ename,sal from emp order by 2;

找出工资在1250和3000之间的员工信息，要求按照薪资降序排列
select ename,sal from emp where sal between 1250 and 3000 order by sal desc;

分组函数(多行处理函数)

多行处理函数特点：多个输入最终输出一行

  1. count 计数
  2. sum 求和
  3. avg 平均值
  4. max 最大值
  5. min 最小值

注意：

  1. 分组函数在使用时必须先进行分组，然后才能使用；如果没有分组，则整张表默认为一组
  2. 分组函数自动忽略 null，不需要提前对 null 处理
  3. 分组函数中 count(*) 和 count(具体字段)的区别
    • count(具体字段) 统计该字段下所有不为 null 的纪录的总数
    • count(*) 统计表的行数
  4. 分组函数不能直接使用在 where 语句中
  5. 所有分组函数可以组合起来一起使用

找出最高工资

select max(sal) from emp;

找出最低工资

select min(sal) from emp;

计算工资和

select sum(sal) from emp;

计算平均工资

select avg(sal) from emp;

计算员工数量

select count(ename) from emp;

找出比最低工资高的员工信息

select ename,sal from emp where sal > min(sal); (×)

组合使用分组函数

select sum(sal), min(sal), max(sal), avg(sal), count(*) from emp;

MySQL 中的数据类型

分类
  • varchar(最长255)
    • 可变长度的字符串，根据实际数据长度动态分配空间
    • 优点：节省空间
    • 缺点：动态分配空间，速度慢
  • char(最长255)
    • 定长字符串，分配固定长度空间存储数据
    • 优点：不需要动态分配空间，速度快
    • 缺点：使用不恰当时，可能造成空间浪费
  • int(最长11)
    • 整型
  • bigint
    • 长整型
  • float
    • 单精度浮点型
  • double
    • 双精度浮点型
  • date
    • 短日期
  • datetime
    • 长日期
  • clob
    • 字符大对象(Character Large OBject)
    • 最多存 4G 的字符串
    • 超过255个字符的都要采用 clob
  • blob
    • 二进制大对象(Binary Large OBject)
    • 存储图片、声音、视频等流媒体数据
    • 往 blob 类型插入数据时，需要使用 IO 流

delete 语句的删除原理

表中的数据被删除了，但是数据在硬盘上的真实存储空间不会被释放

  • 缺点：删除效率低
  • 优点：支持回滚，后悔可以恢复数据
truncate 语句的删除原理

表被一次截断，物理删除

  • 优点：快
  • 缺点：不支持回滚

主键约束

概念

主键值是每行纪录的唯一标识，任何一张表都应该有主键，否则表无效，一张表只能添加一个主键约束

主键特征

not null + unique

实例

建议

主键值建议使用 int/bigint/char 等类型，不建议使用 varchar，主键值一般都是数字，一般是定长的

自然主键/业务主键

  • 自然主键：主键值是一个自然数，与业务无关 (实际开发使用多)
  • 业务主键：主键值与业务紧密相关，例如银行卡号作主键

主键一旦和业务挂钩，那么当业务发生变动时，可能影响到主键值，所以不建议使用业务主键

在 mysql 中，有一种机制可以自动帮助我们维护一个主键值，auto_increment 表示从 1 开始以 1 递增

概念

存储引擎是 MySQL 中特有的术语，其他数据库中一般没有；存储引擎是表存储/组织数据的方式，不同的存储引擎，表存储数据的方式不同

show engines \G

MyISAM

它管理的表具有以下特征：

  • 使用三个文件表示每个表：
    • 格式文件 — 存储表结构的定义（mytable.frm）
    • 数据文件 — 存储表行的内容（mytable.MYD）
    • 索引文件 — 存储表上索引（mytable.MYI）
  • 灵活的 AUTO_INCREMENT 字段处理
  • 可被转换为压缩、只读表来节省空间
  • 不支持事务，安全性低

补充

对于一张表，只要是 主键 或者加有 unique约束 的字段会自动创建索引

InnoDB

  • InnoDB 存储引擎是 MySQL 的缺省引擎。
  • 它管理的表具有下列主要特征：
    • 默认，重量级
    • 支持事务，支持MySQL服务器崩溃后自动恢复
    • 最主要特点是安全
    • 每个 InnoDB 表在数据库目录中以.frm 格式文件表示
    • InnoDB 表空间 tablespace 被用于存储表的内容（表空间是逻辑名称，存储 数据 + 索引）
    • 提供一组用来记录事务性活动的日志文件
    • 用 COMMIT(提交)、SAVEPOINT 及 ROLLBACK(回滚)支持事务处理
    • 提供全 ACID 兼容
    • 多版本（MVCC）和行级锁定
    • 支持外键及引用的完整性，包括级联删除和更新
    • 效率不高，不能压缩，不能转为只读，不能很好地节省存储空间

MEMORY

  • 使用 MEMORY 存储引擎的表，其数据存储在内存中，且行的长度固定，这两个特点使得 MEMORY 存储引擎非
    常快。
  • MEMORY 存储引擎管理的表具有下列特征：
    • 在数据库目录内，每个表均以.frm 格式的文件表示。
    • 表数据及索引被存储在内存中(目的就是快，查询快，缺点：不安全，关机后数据消失)。
    • 表级锁机制。
    • 不能包含 TEXT 或 BLOB 字段。
  • MEMORY 存储引擎以前被称为 HEAP 引擎。

事务

概念

一个事务其实就是一个完整的业务逻辑

注意

只有 DML 语句（insert, delete, update）才有事务，其他的语句与事务无关。

InnoDB 事务引擎：提供一组用来记录事务性活动的日志文件

在事务执行中，每条 DML 的操作纪录都会纪录到 “事务性活动的日志文件” 中。

在事务执行中，可以提交事务、回滚事务

  • 提交事务

清空事务性活动的日志文件，将数据全部彻底持久化到数据库表中，提交事务标志着事务的结束，并且是一种全部成功的结束

  • 回滚事务

将之前所有的 DML 操作全部撤销，也会清空事务性活动的日志文件，回滚事务也标志着事务的结束，并且是一种全都失败的结束

语法

  • 提交事务：commit;
  • 回滚事务：rollback;(回滚永远都只能回滚到上一次的提交点)
  • 事务对应的英文：transaction

注意

MySQL 在默认情况下自动提交事务，每执行一条 DEL 语句则提交一次，其实是不符合开发习惯的

如何将 MySQL 的自动提交机制关闭？

start transaction;

事务四个特性

  • A：原子性(Atomicity)

    事务时最小单元不能再分

  • C：一致性(Consistency)

    在同一个事务中，所有操作必须同时成功，或者同时失败，以保证数据的一致性

  • I：隔离性(Isolation)

    A事务和B事务之间具有一定的隔离

  • D：持久性(Durability)

    事务最终结束的保障，事务提交，就相当于将没有保存到硬盘上的数据保存到硬盘上

事务的隔离性

事务的隔离级别

  • 读未提交：read uncommitted (最低隔离) 《没有提交就读到了》
    • 事务A可以读取到事务B未提交的数据
    • 存在问题：脏读(dirty read)
    • 这是一种理论上的隔离级别，大多数数据库隔离级别都是二档起步！
  • 读已提交：read committed 《提交后才能读到》
    • 事务A只能读取到事务B提交之后的数据
    • 解决了脏读的现象
    • 存在问题：不可重复读取数据
    • 不可重复读：事务开启后，第一次读到的数据是3条，当前事务还未结束，第二次在读取时，读到的数据可能是4条
    • oracle 默认级别
  • 可重复读：repeatable read 《提交后也读不到，读取的永远都是开启事务时的数据》
    • 事务A开启后，不管多久，每次在A中读取到的数据都是一致的。即使事务B将数据已经修改，并且提交了，事务A读取到的数据还是没有发生改变
    • 解决了不可重复读的问题
    • 存在问题：幻读
    • mysql 默认级别
  • 序列化/串行化： serializable (最高隔离)
    • 效率最低，解决了所有问题
    • 表示事务排队，不能并发

索引

概念

  • 索引时在数据库的字段上添加的，一张表的一个字段可以添加一个索引，多个字段联合起来也可以添加索引
  • 在任何数据库中，主键都会自动添加索引；在mysql中，一个字段如果有unique约束，也会自动创建索引
  • 在任何数据库中，任何一张表的任何一条纪录(每一行)在硬盘存储上都有一个硬盘的物理存储编号
  • 在 mysql 中，索引是一个单独的对象，不同的存储引擎以不同的形式存在
    • 在 MyISAM 存储引擎中，索引存储在一个 .MYI 文件中
    • 在 InnoDB 存储引擎中，索引存储在一个逻辑名称为 tablespace 中
    • 在 MEMORY 存储引擎中，索引存储在内存中
    • 不管索引在哪，索引在 MySQL 当中都是以一个树的形式存在(B-Tree)

考虑添加索引的条件

  1. 数据量庞大
  2. 该字段经常出现在 where 后
  3. 该字段很少 DML(update,delete,insert) 操作，因为 DML 后，索引需要重新排序
  4. 建议不要随意添加索引，索引也是需要维护的，太多反而会降低系统的性能，建议通过主键查询，通过 unique 约束的字段查询，效率是比较高的

索引失效

索引失效1

select * from emp where ename like '%T';

ename 上即使添加了索引，也不会走索引，原因是因为模糊匹配中以”%开头，尽量避免模糊查询以”%”开始，这是一种优化手段

索引失效2

使用 or 会失效，如果使用 or，那需要 or 两边的条件字段都要有索引，才会走索引，所以不建议使用 or

索引失效3

使用复合索引时，没有使用左侧的列查找，索引失效

索引失效4

在 where 中，索引列参加了运算，索引失效

索引失效5

在 where 中索引列使用了函数，所以失效

索引分类

  • 单一索引：一个字段上添加索引

  • 复合索引：两个字段或者更多的字段上添加索引

  • 主键索引：主键上添加索引

  • 唯一性索引：具有 unique 约束的字段上添加索引

注意

唯一性比较弱的字段上添加索引用处不大

数据库设计三范式

  1. 任何一张表必须有主键，每一个字段原子性不可再分
  2. 所有非主键字段完全依赖主键，不要产生部分依赖
  3. 在第二范式的基础上，要求所有非主键字段直接依赖主键，不要产生传递依赖

设计数据库表时，按照以上范式进行，可以避免表中的数据的冗余，空间的浪费

多对多怎么设计：多对多，三张表，关系表两个外键

一对多怎么设计：一对多，两张表，多的表加外键

一对一字段太多怎么设计：两张表，另一张表外键加唯一

最后：

数据库设计三范式是理论上的。

实践和理论有的时候有偏差。

最终的目的都是为了满足客户的需求，有的时候会拿冗余换执行速度。

因为在sql当中，表和表之间连接次数越多，效率越低。（笛卡尔积）

有的时候可能会存在冗余，但是为了减少表的连接次数，这样做也是合理的，
并且对于开发人员来说，sql语句的编写难度也会降低。
